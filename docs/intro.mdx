# Introduction

![Heir cluster architecture](/img/Heir-Logo-Lockup-Gold.svg)


Heir is an open source Kubernetes distribution designed to make it easy to operate and run Kubernetes clusters at scale. Unlike traditional distributions that
scatter control plane components across dedicated nodes, Heir collapses the entire control plane into a single stateless pod running inside a Management
Kubernetes Cluster. That management cluster can be any conformant Kubernetes distribution, so each tenant cluster inherits the availability guarantees of
whatever management infrastructure you already run. Operators, add-ons, and cluster controllers all live there, cleanly separated from the workloads they
manage, and because they run centrally, a single operator instance can govern any number of data planes simultaneously, dramatically reducing the total footprint
required to manage large fleets of clusters.

The data plane is a self-contained binary that packages almost everything required to run a Kubernetes worker node, with the exception of the Linux kernel and
its modules. Nodes in the data plane are purpose-built for compute: they run workloads and nothing else, with no co-located operators, no add-on sidecars, and
no control plane processes competing for resources with your applications. This binary can be deployed on bare metal servers, containers, air-gapped
environments, public cloud instances, or any other compute substrate. Because the management cluster owns everything that is not a workload, the blast radius of
a misbehaving operator or add-on is fully contained there and never reaches the nodes running your production workloads.


## Philosophy

Heir takes a different approach to how cluster add-ons and operators are deployed. In many Kubernetes setups, components like monitoring agents, network
controllers, and storage operators run directly on worker nodes, which works well when those components need to be close to the workload. For add-ons that can
run remotely, however, Heir moves them to the management cluster instead, keeping data plane nodes focused on application workloads.

This separation reduces resource contention on worker nodes, simplifies node lifecycle management, and allows a single operator instance to serve multiple data
planes at once. It is a deliberate trade-off suited for environments where centralized management is practical, and it is the foundation on which most of Heir's
operational properties are built.


## Why Heir?

* **Low cost**: minimizes infrastructure overhead by collapsing control plane components into a single stateless pod
* **No 3-node HA tax**: runs a highly available cluster without the traditional minimum node requirement
* **Single artifact**: distributed as a self-contained binary and Docker image with everything needed to run a cluster
* **Operator-friendly**: no deep Kubernetes expertise required to operate at scale
* **Secure by default**: reduced attack surface, as everything Heir needs to run is managed and contained within the management cluster
* **Automated operations**: certificate renewal and cluster upgrades are handled automatically
* **Bring your own everything**: pluggable components let you integrate your existing tools and infrastructure
* **Fleet-scale management**: a single management cluster can govern hundreds of data plane clusters with no additional operator instances
