# Architecture

## Overview

Organizations that adopt Kubernetes at scale typically end up managing many clusters, one per environment, per tier, per team, or per business domain. In the
traditional model, every one of those clusters carries its own control plane: at least three dedicated nodes to satisfy the etcd quorum, plus the operational burden
of keeping each control plane healthy, upgraded, and available independently. Multiply that across dozens of clusters and the cost becomes significant, both in
infrastructure and in engineering time.

Heir takes a different approach. Rather than treating the control plane as a fixed part of each cluster's infrastructure, Heir runs control plane components
as a `single stateless pod` inside a shared management Kubernetes cluster. This shifts the availability problem: platform engineers only need to think about
control plane availability once, when designing the management cluster. Any additional tenant clusters provisioned on top of it inherit that availability
automatically, with no extra infrastructure to provision or maintain.

Because the control plane is just another cloud-native application running on Kubernetes, it behaves like one: it can be stopped, rescheduled, and restarted
without data loss, thanks to an external storage backend. The Kubernetes control plane stops being a seven-headed monster that requires specialist knowledge to
operate and becomes a workload that any Kubernetes operator already knows how to manage.


![Heir cluster architecture](/img/cluster-architecture.svg)

Heir is split into two distinct planes: the `control plane`, which runs inside the management cluster, and the `data plane`, which runs on worker nodes.
The two communicate exclusively over connections initiated by worker nodes, the control plane never reaches out to workers directly.



## Control Plane

The Heir control plane packages the standard Kubernetes control plane components (API server, scheduler, controller manager, and the communication proxy) as a single stateless pod running inside a management Kubernetes cluster. The management cluster can be any conformant Kubernetes distribution, including managed services such as EKS, GKE, AKS, or DOKS.

![Heir control plane](/img/control-plane.svg)

Running as a stateless pod has a fundamental implication: the control plane is no longer a special snowflake that requires dedicated infrastructure and specialist knowledge to keep alive. It is just another workload. Kubernetes already knows how to schedule, restart, and reschedule workloads, and it does the same for Heir control planes. State is never stored in the pod itself; it lives in an external storage backend, so the pod can be terminated and restarted on any node at any time without data loss.

This also means availability is no longer a per-cluster concern. When a new tenant cluster is provisioned, its control plane inherits the availability guarantees of the management cluster without any additional infrastructure. A management cluster backed by a managed Kubernetes service with a multi-zone control plane gives every tenant cluster running on it the same resilience, for free.

### Process Model

All control plane components run as separate processes inside a single container, managed by [S6 Overlay](https://github.com/just-containers/s6-overlay). The container
environment is configured so that no process writes local state (beyond logs), meaning the principle is that faults are always a consequence of not satisfying external dependencies,
it also means the container can be stopped and restarted at any time without consequence.

> One of the oft-repeated Docker mantras is "one process per container", but we disagree. There's nothing inherently bad about running multiple processes in a container.
The more abstract "one thing per container" is our policy - a container should do one thing, such as "run a chat service" or "run gitlab." This may involve
multiple processes, which is fine.

### MasterAgent

The MasterAgent is a process that runs inside every Heir control plane pod. It starts and owns the kine storage process, which provides the API server with an etcd-compatible interface backed by an external database. Alongside kine, it aggregates the health of all Kubernetes control plane components and exposes a readiness endpoint used for health checks by the Management runtime.

The MasterAgent also solves an address resolution problem inherent to running the API server as a pod. When the API server starts inside a management cluster pod, the `kubernetes` Service in the default namespace is set to the cluster-internal address of that pod, which is unreachable by worker nodes that live outside the management cluster. To bridge this gap, the MasterAgent continuously reads control plane addresses from the `worker-profile ConfigMap`, resolves any hostnames to IPs, and updates the EndpointSlice backing the `kubernetes` Service in the default namespace. This keeps in-cluster DNS resolution of `kubernetes.default.svc` accurate as the control plane's network topology changes, without requiring manual intervention or pod restarts.

### Operators and Addons

In traditional Kubernetes, operators and addon control planes (such as Istiod, ingress controllers, or custom controllers) run inside the cluster they manage. This couples
their availability and resource consumption to the data plane, and means each cluster needs its own copy of every addon, even when those addons are functionally
identical across clusters and support multi-tier.

Heir inverts this: **operators and addons run in the management cluster**, communicating with tenant clusters through their API servers. An ingress controller or service mesh control plane deployed in the management cluster can manage resources across multiple data planes simultaneously, with no footprint on the worker nodes themselves.

This separation has meaningful consequences:

- Worker nodes are reserved exclusively for application workloads
- A single operator instance can govern multiple data planes simultaneously (if supported), reducing duplication
- Addon vulnerabilities and bugs are contained within the management cluster and do not affect running workloads
- Addons can be upgraded centrally without touching worker nodes



## Data Plane

The Heir data plane is distributed as a `single self-contained binary` that packages everything needed to run a Kubernetes worker node, except the Init System, iptables, and Linux kernel modules. It runs on bare metal, virtual machines, containers, or any Linux host.

### Worker Proxy

Worker nodes connect to the tenant cluster through a local reverse proxy running on the loopback interface, started as part of the `heir worker` process. The proxy maintains a dynamic list of control plane endpoints and load-balances all outbound API server traffic across them.

Initially, the proxy is seeded with the addresses embedded in the `bootstrap kubeconfig` provided at provisioning time. Once the node has completed bootstrap and can authenticate to the cluster, it reads the `worker-profile ConfigMap` from the `kube-system` namespace. This ConfigMap is continuously updated by the Heir controller and contains the definitive list of control plane addresses; the proxy's endpoint pool is refreshed from this source, allowing nodes to track control plane topology changes without reconfiguration.

The MasterAgent running inside the control plane pod mirrors this by keeping the Kubernetes EndpointSlice in the default namespace in sync with the same addresses, ensuring that in-cluster DNS resolution of `kubernetes.default.svc` remains accurate as infrastructure changes.

### Worker Profile

The `worker-profile ConfigMap` in `kube-system` is the single source of configuration distributed to all worker nodes. It carries:

- Control plane endpoint addresses
- A `KubeletConfiguration` document
- Additional kubelet flags
- CNI provider details

The worker process caches the profile locally after reading it, so kubelet can be restarted cleanly even if the API server is temporarily unreachable.
The `heir worker` process continuously watches this ConfigMap for changes, it is this reconciliation loop that enables automatic cluster
operations such as kubelet configuration updates and CNI reconfiguration without manual intervention or node re-provisioning. Any update to the
Runtime spec that affects worker configuration is reflected here and picked up by nodes on their next startup.

### Bootstrap Tokens

Nodes register using a short-lived bootstrap token generated by an administrator with the Heir CLI. The token grants its bearer the minimum RBAC permissions
necessary to submit a Certificate Signing Request. Once TLS bootstrap completes, the token is no
longer used and the node authenticates exclusively with its signed client certificate.

When `heir worker` starts, it uses the bootstrap token to submit a CSR to the API server requesting a kubelet client certificate identifying the
node as `system:node:[hostname]`. The MasterAgent watches for pending CSRs and approves them only after verifying that every SAN in the
request matches an address already registered on the corresponding Node object. This prevents a compromised token from being used to obtain
certificates for arbitrary node identities.




## Network Model

All connections flow **from worker nodes to the control plane**. The control plane never initiates connections to workers. Requests from the control plane to workloads on workers (such as `kubectl exec` or log streaming) are tunneled over the same outbound connection established by the worker through multiplexing.

This design has practical consequences:

- Worker nodes can sit behind firewalls or NAT with no inbound rules required
- Heir works in **air-gapped environments** without internet access after initial setup
- Heir is well suited to **edge computing** deployments where workers are in remote or constrained networks
- Control planes can run on managed cloud Kubernetes services while workers run anywhere



## Multi-tenancy

Because each tenant cluster is a single pod in the management cluster, running multiple tenant clusters is as simple as deploying multiple `Runtime` resources. Each gets its
own isolated control plane, API server, and kubeconfig, while sharing the management cluster's infrastructure. This makes Heir an efficient model for multi-tenant Kubernetes at
scale.
